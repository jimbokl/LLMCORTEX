# The ratio: how Cortex caught its own blind spot in audit logs

*2026-04-11 · ~800 words*

I built a Claude Code hook that injects lessons from past failures into an agent's context before it reasons about a task. I expected my hand-written YAML rules to carry most of the load. Five days after deployment, the audit log told me I was wrong — and the way it told me is the story.

## Setup

Cortex is a `UserPromptSubmit` hook for Claude Code. When you submit a prompt, a 60-ms subprocess classifies the text against YAML rules, fetches matched tripwires (structured lessons from past failures), composes them via a cost-component synthesizer, and writes a `<cortex_brief>` block into the agent's additional context. The agent then reasons about your task with the brief in its working memory, before it decides to run any tool.

The rule engine is simple: each rule has `match_any` and `and_any` keyword sets. A rule fires when both sets intersect the prompt tokens. I hand-wrote 10 rules over two afternoons. They seemed comprehensive for the research project I was working on (Polymarket trading). In testing, they caught the prompts I expected them to catch.

## Day 4: the miss

Five days into development, a collaborator asked me a question that went something like *"show me the top features ranked by PnL"*. The Cortex hook fired, matched zero rules, and silently returned. The agent (me, in this case — the tool was live in my own Claude Code sessions) proceeded to answer the question without any brief.

That should have been fine. The question is legitimate. But I happened to check the session log right after, and the absence of any `inject` event made me pause. The prompt was clearly in domain. It mentioned PnL. It was about the Polymarket project. My rules should have caught it.

They didn't. Why?

Because my `poly_fee_pnl` rule required the token `pnl` **and** one of `[poly, polymarket, clob, backtest, replay]`. The prompt had `pnl` but none of the domain anchors — the collaborator referred to the topic by shared context, not by explicit naming. The rule engine saw a non-matching prompt and stayed silent.

I spent an afternoon building what I initially thought was a quick patch: a TF-IDF fallback that scores tripwire bodies against prompt tokens when the primary rule engine returns zero matches. Weighted overlap: trigger match = 3.0, title match = 3.0, body match = 1.0. Top-3 above a threshold get injected. 130 lines of pure Python. Day 4's work was this fallback plus tests.

## Day 9: the audit log receipts

Five days after Day 4, I ran `cortex stats --sessions` on the live audit log and saw the actual breakdown. In 17 active sessions across 13 days of development:

- **20 primary rule-engine injections**
- **21 keyword-fallback injections**
- **16 synthesizer fires**
- **1 potential violation logged** (a deliberate test case)

The fallback is firing essentially **as often as the primary rule engine**. In sessions where the hook produced any brief at all, **10 of 17 relied on the fallback exclusively**. Only 6 used the primary rules, and 1 used both.

This surprised me. When I built Cortex, the rules were "the product" and the fallback was "a safety net for edge cases". The audit log told me the fallback is not a safety net. It's load-bearing.

## Why this is the story

I could pattern-match this onto a dozen generic engineering lessons about measurement, premature abstraction, or the limits of hand-written rules. But the actually interesting thing is narrower:

**Cortex caught its own blind spot using the same audit mechanism it uses to catch agent failures.**

The whole design of Cortex is built around writing every hook invocation to a session log so Day-6 silent-violation detection can later count which lessons got applied vs. ignored. That's the "active instinct" pipeline — measure what the hook did, compare to what the agent did afterward, feed the discrepancy back into a better rule set.

I built that pipeline to monitor agents. It ended up monitoring me first, because my rule-engine design was the first thing in the loop to be inadequate. The audit log showed me the rules were too narrow before any agent failure did. The Day 4 TF-IDF fallback exists because the audit log told me, not because I anticipated it.

That's a feedback loop I didn't explicitly design for. It came for free because I chose to log every hook invocation and every tool call.

## What this means architecturally

Three things fall out of this for anyone building similar systems:

1. **Log the hook itself, not just what the hook did.** If your memory layer is silent on some prompts, you want to know. `inject` events are easy to count; the absence of one is just as informative.

2. **TF-IDF over curated bodies beats semantic search over everything.** I tried the semantic-search version too (see the Day-4 Palace-daemon post-mortem in the repo). It had worse coverage and 3× the infrastructure cost. A 130-line weighted token scorer over ~15 short tripwire bodies does 50% of the load-bearing work in production.

3. **Hand-written rules age worse than curated data.** The tripwires themselves (the lessons) are stable — a past failure is a past failure. The *triggers* that route tasks to tripwires are what rots. New prompt shapes appear that you didn't anticipate, and the rules need widening, tightening, or retiring. That's what `cortex suggest-patterns` and `cortex reflect` were built to automate.
